开门红bug:

在merge lab3 之后，再次运行make qemu是出现一下错误：

用gdb发现kern_pgdir在boot_alloc()之后还是好好的，但在memset后被置0x0了，迷之bug。

问题解决参考：

https://qiita.com/kagurazakakotori/items/334ab87a6eeb76711936

简单来说，要研究boot_alloc()中, end[]是否真的是bss的末尾。 博客中发现，在gcc 7之后，编译的kernel的bss段中，kern_pgdir落在了变量在end[]后面！淦, 就是说kern_pgdir这个指针的地址，刚好在boot_alloc(PGSIZE) 创建的这段区域中。一调用memset(kern_pgdir)就把pde_t* kern_pgdir这个指针给置零了。这里是个链接器的坑:

The procedure is to link all .bss section of kernel objects to kernel, then set the end symbol. However, the output objdump -t obj/kern/pmap.o | grep kern_pgdir shows kern_pgdir is actually in COMMON section. In my personal opinion, the COMMON will be linked implicitly to .bss after end is set. To solve the problem, COMMON should be linked explicitly to .bss before end is set. That’s why we need to modify the linker script as above.

只用将kernel.ld中的bss段显式加上COMMON的链接即可。

ELF文件结构回顾

  • link address: 程序链接的地址，指的是程序的逻辑地址空间。
  • load address: 是程序二进制代码加载的地址，在加载时确定。

之所以区分这两个地址，主要是内核在加载ELF文件时要先把load address加载到内核空间地址中的一片空间暂存，然后配置新的用户页表，为程序提供运行的“逻辑地址空间”环境，也就是link address。也就是说，link address和load address实际上对应的是两张页表。（待证实）
